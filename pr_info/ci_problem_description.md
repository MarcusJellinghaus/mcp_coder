**Failure 1 — `test_verify_config_valid_with_all_sections` (line 133):** The test asserts that all entries have status "ok" or "info", but one entry has status "warning". The root cause is that when `verify_config()` processes the `[coordinator]` schema section (which only defines `cache_refresh_minutes`), `_get_section_data()` returns the full `coordinator` dict from TOML, which includes a `repos` sub-table. The fix is to exclude keys that correspond to sub-tables handled by the wildcard `coordinator.repos.*` pattern — either by filtering out dict-valued keys in the unknown-key loop, or by adding `repos` awareness to the coordinator section check.

**Failures 2 and 3 — `test_verify_config_env_var_only` (line 168) and `test_verify_config_mcp_configured_via_env_var` (line 318):** Both tests set an environment variable (`GITHUB_TOKEN` and `MCP_CODER_MCP_CONFIG` respectively) with an empty config file and expect `verify_config()` to report the field as "ok" with "(env var)" in the value. The fix is to still walk the section's schema fields when the section is absent from the config file, checking each field's `env_var` for a set environment variable, and reporting "ok" with "(env var)" when found. The simplest approach: when `section_data` is `None`, pass an empty dict `{}` to `_verify_section()` instead of skipping it entirely.
