The CI unit-tests job fails with 4 test failures, all in verify command tests that expect `execute_verify()` to return exit code 0 but receive exit code 1. The failures are in `test_verify_command.py::test_execute_verify_returns_zero_on_success`, `test_verify_orchestration.py::test_mcp_import_error_shows_info_message`, `test_verify_orchestration.py::test_mcp_failure_informational_for_claude`, and `test_verify_resolve_llm.py::test_verify_defaults_to_claude`.

The root cause is that Step 3 of the implementation plan (issue #748) added Claude MCP server verification via `parse_claude_mcp_list()` to `execute_verify()`, but the 4 failing tests were not updated with the necessary mocks for this new code path. When `mcp_config_resolved` is truthy, `execute_verify()` now calls `prepare_llm_environment()` and `parse_claude_mcp_list()`. Since these functions are not mocked in the failing tests, they either run against real (unavailable) infrastructure or return `None`. Per the implementation's Decision 12, when `parse_claude_mcp_list` returns `None` and the active provider is `claude`, `claude_mcp_ok` is set to `False`, which causes `_compute_exit_code()` to return 1.

Specifically: in `test_verify_command.py` and `test_verify_resolve_llm.py`, `resolve_mcp_config_path` is not mocked, so it auto-discovers `.mcp.json` in the working directory, entering the MCP verification branch unexpectedly. In `test_verify_orchestration.py`, `resolve_mcp_config_path` IS mocked (returning `/fake/.mcp.json`), but `parse_claude_mcp_list` and `prepare_llm_environment` are not mocked, so the Claude MCP check runs unmocked and fails. The captured CI output confirms this — it shows MCP server errors like `[Errno 2] No such file or directory: '${MCP_CODER_VENV_PATH}\\mcp-tools-py.exe'` and `connection refused`.

The fix requires adding mocks to the 4 failing tests in these files: `tests/cli/commands/test_verify_command.py`, `tests/cli/commands/test_verify_orchestration.py`, and `tests/cli/commands/test_verify_resolve_llm.py`. Each test needs either (a) a mock for `parse_claude_mcp_list` that returns a list of `ClaudeMCPStatus` objects with `ok=True`, or (b) a mock for `resolve_mcp_config_path` returning `None` to skip the MCP branch entirely (where appropriate). For the two orchestration tests that explicitly test MCP behavior with `mcp_config=".mcp.json"`, option (a) is correct — mock `parse_claude_mcp_list` to return successful statuses so the Claude MCP check passes. For the command and resolve_llm tests that don't test MCP at all, option (b) — mocking `resolve_mcp_config_path` to return `None` — is the cleaner approach.
